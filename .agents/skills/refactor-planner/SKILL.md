# Refactor Planner

対象コードを分析し、具体的で実行可能なリファクタリング計画を立案して `plan/refactor/` に出力する。

## いつ使うか

以下のケースで発動する:
- ユーザーが「リファクタリング」「整理」「改善」「再設計」を求めている
- コードの可読性・保守性・テスタビリティの向上を目的とした変更
- モジュール分割、責務の分離、依存関係の整理
- 技術的負債の解消や設計パターンの適用
- 「このまま拡張すると辛い」という将来的な懸念への対処

以下のケースでは使わない:
- バグ修正（動作を変えずに構造だけ変えるのがリファクタリング）
- 新機能の追加（task-splitter スキルを使う）
- 単純な変数名の変更など1ステップで完了する修正

## リファクタリング計画立案のプロセス

### Step 1: 対象コードの分析

ユーザーの意図とスコープを把握してから、関連するコードを読む:

1. **スコープの確認**: 何を対象にするか（特定のファイル・モジュール・全体）
2. **コードの読み込み**: GlobとGrepで対象ファイルを特定し、Readで内容を確認
3. **問題の特定**: 以下の観点から現状の課題を洗い出す

**分析の観点**:
- **単一責任原則の違反**: 1つのクラス/関数が複数の役割を担っていないか
- **長すぎるファイル・関数**: 認知負荷が高く修正が難しい
- **重複コード**: DRY原則に反する繰り返し
- **依存関係の複雑さ**: 循環依存、密結合、テストしにくい構造
- **命名の不一致**: 意図が伝わらない名前や一貫性のないスタイル
- **マジックナンバー・ハードコード**: 設定値がコードに埋め込まれている
- **抽象化の欠如または過剰**: 適切なインターフェース・型定義がない

コードを読まずに計画を立てると的外れな提案になる。必ず実際のコードを確認してから分析する。

### Step 2: リファクタリング計画の立案

分析結果をもとに、以下の構造で計画を立案する:

**優先度の決め方**:
- **高**: バグの温床になっている、テストが書けない、理解に時間がかかる
- **中**: 拡張時に困る、コードの重複がある、命名が分かりにくい
- **低**: 見た目の整理、ベストプラクティスへの統一

**計画の粒度**:
- 各ステップは独立して実行・確認できる大きさにする
- 一度に全部変えようとしない（ステップを分けることでリスクを下げる）
- リファクタリング中も既存テストがパスし続けられる順序にする

### Step 3: 計画ファイルの出力

計画を `plans/refactor/` ディレクトリに Markdown ファイルとして出力する。

**ファイル名の規則**:
```
plans/refactor/<YYYY-MM-DD>-<対象モジュール名またはテーマ>.md
```

例:
- `plans/refactor/2025-04-08-job-service-split.md`
- `plans/refactor/2025-04-08-dependency-cleanup.md`

`plans/refactor/` ディレクトリが存在しない場合は作成する。

**計画ファイルのテンプレート**:

```markdown
# リファクタリング計画: <テーマ>

作成日: <YYYY-MM-DD>
対象: <対象ファイル・モジュール>
目的: <なぜリファクタリングするか、何を改善するか>

---

## 現状の課題

<コード分析で特定した問題点を箇条書きで記述>

- **課題1**: <具体的な問題。どのファイルのどの部分か>
- **課題2**: <同上>

---

## リファクタリング方針

<全体的なアプローチと目指す状態を説明>

---

## 実施ステップ

### Step 1: <最初のステップのタイトル>

**優先度**: 高 / 中 / 低
**対象ファイル**: `src/core/xxx.py`
**作業内容**:
- <具体的に何をするか>
- <どう変えるか>

**完了条件**:
- <何をもってこのステップが完了とするか>
- <既存テストがパスすること、など>

### Step 2: <次のステップ>

...（以降同様）

---

## 注意点・リスク

<リファクタリング時に気をつけるべきこと、破壊的変更になりうる箇所など>

---

## 完了後の期待効果

<リファクタリング後にどう良くなるか>
```

### Step 4: ユーザーへの報告

計画ファイルを出力したら:
1. 出力先のファイルパスをユーザーに伝える
2. 計画の概要（課題・ステップ数・主要な変更点）をサマリーとして提示する
3. 「このまま実施しますか？」と確認を取る（task-splitter で実施に移れることも伝える）

## 出力例のサマリー形式

```
## リファクタリング計画を作成しました

**出力先**: `plan/refactor/2025-04-08-job-service-split.md`

**特定した課題**:
- `job_service.py` が500行を超え、実行管理・通知・ログの3責務を持っている
- 通知ロジックがサービス層に直接依存しており、テストしにくい

**計画の概要** (3ステップ):
1. 通知ロジックを `NotificationService` として分離（高優先度）
2. ログ管理を専用クラスに切り出し（中優先度）
3. `job_service.py` のリファクタリング後にテストを整備（中優先度）

実施する場合はお知らせください。task-splitter でサブタスクに分解して進めることができます。
```

## 注意点

- **動作を変えないこと**: リファクタリングは外部から見た振る舞いを変えない。機能追加は含めない
- **段階的に進めること**: 一度に全部変えようとせず、ステップを小さく区切る
- **テストが先**: テストのないコードをリファクタリングする場合は、先にテストを書くことを計画に含める
- **ユーザーの優先度を尊重**: 全部やる必要はない。ユーザーが今困っている部分を優先する
