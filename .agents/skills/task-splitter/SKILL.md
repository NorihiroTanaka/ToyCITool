# Task Splitter

ユーザーから受け取った開発タスクを分析し、実行可能なサブタスクに分割して、TodoWriteツールで管理できる形にする。

## いつ使うか

以下のようなケースで発動する:
- ユーザーが複数ステップを要する機能追加を依頼した
- バグ修正で調査・修正・テストなど複数段階が必要
- リファクタリングで複数ファイルにまたがる変更が必要
- 「〜を実装して」「〜を作って」など、漠然とした大きなタスク

以下のケースでは使わない:
- 1ステップで完了する単純な修正
- 質問への回答やコードの説明
- 明確に1箇所だけの変更

## 分割のプロセス

### Step 1: タスクの理解

まずタスクの全体像を把握する。必要に応じて:
- 関連するコードベースを調査する（Glob, Grep, Readツールで）
- 既存のアーキテクチャや依存関係を確認する
- 不明点があればユーザーに質問する（AskUserQuestionツールで）

タスクの規模感と技術的な複雑さを見極めてから分割に進む。コードを読まずに分割すると的外れなサブタスクになりやすい。

### Step 2: サブタスクへの分割

以下の原則に従って分割する:

**分割の粒度**
- 各サブタスクは1つの明確なゴールを持つ（「認証ミドルウェアを作成する」はOK、「いろいろ修正する」はNG）
- 各サブタスクは独立して完了を確認できる
- 多すぎず少なすぎず: 3〜8個程度が目安。2個以下なら分割不要、10個以上なら粒度が細かすぎる可能性がある

**順序と依存関係**
- 依存関係を考慮して自然な実行順序に並べる
- 基盤となる変更（型定義、データモデルなど）を先に
- UIやテストなど依存する作業は後に

**サブタスクの構造**
各サブタスクには以下を含める:
- **何をするか**: 具体的で実行可能なアクション（命令形）
- **activeForm**: 進行中の表現（「〜している」形式）
- 必要に応じて、対象ファイルや変更内容の補足

**典型的な分割パターン**

機能追加の場合:
1. データモデル/型定義の追加
2. ビジネスロジック/サービス層の実装
3. API/インターフェース層の実装
4. UI/表示層の実装（該当する場合）
5. テストの作成・実行
6. 統合テスト・動作確認

バグ修正の場合:
1. 原因の調査・特定
2. 修正の実装
3. テストの追加・修正
4. 関連箇所への影響確認

リファクタリングの場合:
1. 現状の把握と変更計画の整理
2. 構造変更の実装
3. 関連コードの更新
4. テストの更新・実行

これらはあくまでテンプレートであり、実際のタスクに合わせて柔軟に調整する。

### Step 3: ユーザーへの提示と確認

分割結果をマークダウンの番号付きリストで表示する:

```
このタスクを以下のサブタスクに分割しました:

1. **〇〇を作成する** - 補足説明
2. **△△を実装する** - 補足説明
3. **□□のテストを追加する** - 補足説明

この内容でよろしいですか？変更したい点があればお知らせください。
```

AskUserQuestionツールを使って確認を取る:
- 「この分割で進めてよいですか？」
- 選択肢: 「はい、この内容で進める」「修正したい点がある」

### Step 4: TodoWriteへの登録

ユーザーの承認を得たら、TodoWriteツールでサブタスクを登録する。

- 最初のサブタスクを `in_progress` に設定
- 残りは `pending` に設定
- `content` には命令形（「〇〇を作成する」）
- `activeForm` には進行形（「〇〇を作成中」）

登録後、すぐに最初のサブタスクの作業を開始する。

## 注意点

- コードベースを読んでから分割すること。推測だけで分割すると、存在しないファイルや不要な作業が含まれてしまう
- ユーザーが既に明確なステップを指定している場合は、それを尊重する。勝手に再構成しない
- 分割はあくまで手段であり、ユーザーの本来の目的（タスクの完了）を忘れない
