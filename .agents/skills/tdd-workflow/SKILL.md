# TDD Workflow

テスト駆動開発（TDD）の Red-Green-Refactor サイクルを、専用サブエージェントを使いながら実行するスキル。

## 使用モデル

| フェーズ | モデル | 理由 |
|---------|--------|------|
| Step 1: 作業リストの洗い出し | **Opus** | タスク全体を俯瞰し、漏れなく作業を特定する計画立案フェーズ |
| Step 2: テスト作成 | **Sonnet** | テストコードの実装作業 |
| Step 3: テストのコードレビュー | **Sonnet** | コードレビューの実施作業 |
| Step 5: 実装 | **Sonnet** | ソースコードの実装作業 |
| Step 7: リファクタリング | **Sonnet** | コード改善の実施作業 |
| Step 11: 最終コードレビュー | **Sonnet** | コードレビューの実施作業 |

## 全体の流れ

```
タスク受け取り
  ↓
[Opus] 作業リストの洗い出し（TodoWrite）
  ↓
各作業に対して繰り返し:
  ├─ [Sonnet] テスト作成エージェント → テストコードを書く
  ├─ [Sonnet] コードレビューエージェント → テストをレビュー
  ├─ テスト実行（RED確認）
  ├─ [Sonnet] 実装エージェント → テストをパスさせる
  ├─ テスト実行（GREEN確認）
  ├─ [Sonnet] リファクタリングエージェント → コードを整理
  └─ テスト実行（GREEN維持確認）
  ↓
[Sonnet] 最終コードレビューエージェント
```

---

## Step 1: 作業リストの洗い出し（**Opus** で実施）

タスクを受け取ったら、**Opusエージェント**を起動して作業リストを洗い出す。
計画立案は全体を俯瞰した判断が必要なため、高性能モデルを使う。

Opusエージェントへの指示例:
```
以下のタスクを実施するにあたって、必要な作業を洗い出してください。

## タスク
[ユーザーから受け取ったタスクの説明]

## コードベース情報
[関連ファイルの一覧・既存の設計情報]

## 出力形式
作業を実施順に番号付きリストで出力してください。
各作業は「1つのテストで検証できる」粒度で分割してください。
```

Opusエージェントの回答を受け取ったら:
1. 関連ファイルを Glob/Grep/Read で確認する（Opusエージェントに渡す情報を収集するため先に調査してもよい）
2. Opusエージェントが洗い出した作業リストを TodoWrite で全件 `pending` 状態で登録する
3. 最初の作業を `in_progress` にして次のステップへ進む

**作業リストの粒度の目安:**
- 1つの作業 = 1つのテストで検証できる程度の単位
- 「ユーザーを作成できる」「パスワードを検証できる」など、1機能1テスト
- 大きすぎる場合は後から分割（Step 10 で対応）

---

## Step 2〜8: Red-Green-Refactor サイクル

現在の作業を TodoWrite で `in_progress` にしてから開始する。

### Step 2: テスト作成（**Sonnet** サブエージェント）

**テスト作成エージェント**を Agent ツールで起動する。`model: "sonnet"` を指定する。

エージェントへの指示に含める情報:
- 現在の作業内容（何をテストするか）
- 対象ファイル・クラス・関数（コードベース調査済みの情報）
- 使用するテストフレームワーク（プロジェクト内の既存テストを確認して合わせる）
- テストファイルの保存先パス
- **「テストのみ作成し、実装はしないこと」** を明示する

テスト作成エージェントのプロンプト例:
```
以下の作業に対するテストコードを作成してください。

## 作業内容
[現在の作業の説明]

## テスト対象
- ファイル: [対象ファイルパス]
- クラス/関数: [対象の名前]

## テストフレームワーク
[プロジェクトで使用しているフレームワーク]

## テストファイルの保存先
[テストファイルのパス]

## 制約
- テストコードのみ作成してください
- 実装コードは作成・変更しないでください
- テストは現時点で失敗する（RED）ことが期待されます
- 1つの作業に対して必要十分なテストケースを作成してください
```

### Step 3: テストのコードレビュー（**Sonnet** サブエージェント）

**コードレビューエージェント**を起動する。`model: "sonnet"` を指定する。

レビューの観点:
- テストは作業の意図を正確に検証しているか
- テストケースに漏れはないか（正常系・異常系・境界値）
- テストコードの可読性・保守性
- テストの独立性（他のテストに依存していないか）
- 命名規則の一貫性

コードレビューエージェントのプロンプト例:
```
以下のテストコードをレビューしてください。

## 作業内容
[現在の作業の説明]

## レビュー対象ファイル
[テストファイルのパス]

## レビューの観点
1. テストが作業の意図を正確に検証しているか
2. テストケースの網羅性（正常系・異常系・境界値）
3. テストコードの可読性・保守性
4. テストの独立性
5. 命名規則の一貫性

レビュー結果と、必要であれば修正案を提示してください。
```

レビューで重大な問題が指摘された場合は、テスト作成エージェントに修正を依頼する。

### Step 4: テストが RED になることを確認

テストを実行して、**失敗することを確認する**。

```bash
# プロジェクトのテスト実行コマンドで対象テストを実行
```

- テストが失敗（RED）→ 正常。Step 5 へ進む
- テストがパス（GREEN）→ 異常。テストが実装済みの動作を検証しているだけの可能性がある。テスト内容を再確認する
- テストがエラー（コンパイルエラーなど）→ 許容。未実装のクラス/関数を呼び出しているためのエラーであれば正常な RED 状態とみなす

### Step 5: 実装（**Sonnet** サブエージェント）

**実装エージェント**を起動する。`model: "sonnet"` を指定する。

エージェントへの指示に含める情報:
- 現在の作業内容
- テストファイルのパス（テストを必ず読んでから実装する）
- 実装対象ファイルのパス
- **「テストをパスさせることのみに集中すること」** を明示する
- 過剰実装（次の作業まで先に実装するなど）をしないよう指示する

実装エージェントのプロンプト例:
```
以下のテストをパスさせる実装を行ってください。

## 作業内容
[現在の作業の説明]

## テストファイル
[テストファイルのパス]

## 実装対象ファイル
[実装ファイルのパス]

## 制約
- まずテストコードを必ず読んでから実装を始めてください
- テストをパスさせることだけに集中してください
- テストに定義されていない機能を先回りして実装しないでください
- 最小限の実装でテストをグリーンにしてください
```

### Step 6: テストが GREEN になることを確認

テストを実行して、**パスすることを確認する**。

```bash
# テスト実行
```

- テストがパス（GREEN）→ 正常。Step 7 へ進む
- テストが失敗のまま → 実装エージェントに再度修正を依頼する。エラーメッセージを渡して原因を特定させる

### Step 7: リファクタリング（**Sonnet** サブエージェント）

**リファクタリングエージェント**を起動する。`model: "sonnet"` を指定する。

エージェントへの指示に含める情報:
- 実装ファイルとテストファイルのパス
- **「テストをグリーンに保ちながら」リファクタリングすること** を明示する
- コードの重複排除、命名改善、構造整理など

リファクタリングエージェントのプロンプト例:
```
以下の実装コードをリファクタリングしてください。

## 対象ファイル
- 実装: [実装ファイルのパス]
- テスト: [テストファイルのパス]

## リファクタリングの観点
- コードの重複排除
- 変数・関数の命名改善
- 関数の責務を明確にする（1関数1責務）
- 不要なコードの削除
- 可読性の向上

## 制約
- テストはグリーンのまま維持してください
- 機能を変更しないでください（純粋なリファクタリングのみ）
```

### Step 8: テストが GREEN のままであることを確認

テストを実行して、**リファクタリング後もパスすることを確認する**。

```bash
# テスト実行
```

- テストがパス（GREEN）→ 正常。現在の作業を TodoWrite で `completed` にする
- テストが失敗 → リファクタリングで機能が壊れた。リファクタリングエージェントに修正を依頼する

---

## Step 9〜10: 次の作業へ

### Step 9: 残りの作業を確認

TodoWrite の一覧を確認する。
- `pending` の作業が残っている → 次の作業を `in_progress` にして Step 2 から繰り返す
- 全て `completed` → Step 11 へ進む

### Step 10: 作業リストへの追加

実装中に新しく必要な作業が判明した場合:
- TodoWrite に新しい作業を `pending` で追加する
- 適切な順序で挿入する（依存関係を考慮）
- 現在の作業が完了したら、追加した作業も実施する

---

## Step 11: 最終コードレビュー（**Sonnet** サブエージェント）

全作業完了後、**最終コードレビューエージェント**を起動する。`model: "sonnet"` を指定する。

最終レビューの観点:
- 実装全体の一貫性・整合性
- テストの網羅性（全作業に対するテストがあるか）
- コードの品質（可読性・保守性・パフォーマンス）
- セキュリティ上の問題
- ドキュメント（コメント・型アノテーションなど）

最終コードレビューエージェントのプロンプト例:
```
以下の実装全体をコードレビューしてください。

## 実施したタスク
[タスクの説明]

## 変更・作成したファイル
[変更ファイルの一覧]

## レビューの観点
1. 実装全体の一貫性・整合性
2. テストの網羅性
3. コードの品質（可読性・保守性）
4. セキュリティ上の問題
5. 改善提案（必須ではないが推奨される点）

指摘事項があれば優先度（必須修正 / 推奨 / 提案）と合わせて提示してください。
```

レビューで **必須修正** が指摘された場合:
- 対応する作業を TodoWrite に追加
- 必要であれば該当箇所の TDD サイクルをやり直す

---

## 注意事項

### サブエージェントの活用方針

- **テスト作成・実装・コードレビュー** は必ず別エージェントで実施する
- 1エージェントにテストと実装の両方をさせない（TDD の意図が失われるため）
- エージェントへの指示には常に **制約** を明記して、スコープ外の作業をさせない

### モデルの指定方法

Agent ツール呼び出し時に `model` パラメータで指定する:

```
# 計画立案（Step 1）
Agent(model="opus", prompt="...")

# 実装系（Step 2, 3, 5, 7, 11）
Agent(model="sonnet", prompt="...")
```

- **Opus**: 作業リストの洗い出しのみ。タスク全体を俯瞰して漏れなく作業を特定するために使う
- **Sonnet**: テスト作成・実装・コードレビュー・リファクタリング。コスト効率よく実装作業をこなす

### テスト実行はメインエージェントが行う

- テストの実行（RED/GREEN の確認）はメインエージェントが直接行う
- テスト結果に応じて次のステップを判断する

### プロジェクトに合わせる

- テストフレームワーク、テストファイルの命名規則、ディレクトリ構成はプロジェクトの既存コードに従う
- 既存テストがなければ、プロジェクトの言語・フレームワークに適したものを選ぶ

### サイクルの途中で詰まった場合

- 実装エージェントが同じエラーを繰り返す場合は、エラーメッセージと実装コードを添えてより詳細な指示を出す
- テストが RED にならない場合は、テスト自体の見直しを検討する
- 不明点はユーザーに確認する（AskUserQuestion）
